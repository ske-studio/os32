# OS32 開発ポリシー

OS32カーネルおよび外部プログラム開発における公式ポリシー。
コーディング規約、ビルド手順、バージョン管理、テスト、リリースに関する統一ルールを定める。

技術仕様・アーキテクチャの詳細は [DEVELOPMENT.md](DEVELOPMENT.md) を参照。
デバッグに関するポリシーは [POLICY_DEBUG.md](POLICY_DEBUG.md) を参照。

---

## §1. 基本方針

### 設計哲学

- **ホビープロジェクトとしての品質優先**: デッドラインよりも設計品質・コードの可読性を重視する。
- **学習と記録**: 過去のバグや設計判断は必ずドキュメントに残し、同じ問題を二度踏まないようにする。
- **段階的な進化**: 動作する最小単位を積み上げていく。大規模な書き直しよりも、小さな改善の積み重ねを好む。

### AI との協調開発

- AI コーディングアシスタント (Cursor, Gemini 等) を積極的に活用する。
- ただし、AI 生成コードについては必ず人間がレビューし、ハードウェア制約への違反がないか確認する。
- AI 向けの指示書 (`GEMINI.md` 等) はバージョン管理し、ルール変更時は指示書も同時に更新する (Docs as Code)。

## §2. コーディング規約

### C89 (GNU89) 厳守

GCC 環境においても `-std=gnu89` を維持する。

| ルール | 例 |
|--------|-----|
| `//` コメント禁止 | `/* コメント */` を使用 |
| 変数宣言はブロック先頭のみ | `for (int i = 0; ...)` は禁止 |
| C99以降の構文は使用しない | `_Bool`, `restrict`, 可変長配列等は禁止 |

### 呼び出し規約

- **カーネル/外部プログラム**: System V i386 ABI (`-m32`)
- **KernelAPI**: 外部公開関数は必ず `kapi/` に `__cdecl` ラッパーを作成し、スタック渡しでブリッジする

### 文字列操作

カーネル内での文字列操作は `lib/kstring.h` の関数を優先して使用すること。

| 関数 | 用途 |
|------|------|
| `kstrncpy` | 安全な文字列コピー |
| `kstrncat` | バッファサイズを考慮した安全な文字列連結 |
| `kstrlen` | 文字列長取得 |
| `kstrcmp` / `kstrncmp` | 文字列比較 |

### 定数定義の3層管理

| レイヤー | 定義場所 | 例 |
|---------|---------|-----|
| Shared Layer | `os32_kapi_shared.h` | KAPI構造体メンバ名、エラーコード |
| Subsystem Layer | 各サブシステムヘッダ (`vfs.h`, `pc98.h` 等) | `VFS_MAX_PATH`, `TVRAM_COLS` |
| Local Layer | 各モジュール内で定数化 | バッファオフセット等 (ハードコードの `500` 等は禁止) |

## §3. ハードウェア制約

### EGC / GRCG / GDC 使用禁止

**ハードウェアアクセラレータ (EGC / GRCG / GDC描画コマンド) は一切使用しない。**
全てのピクセル操作は CPU (i386) による VRAM (`0xA8000`) 直接書き込みで行う。

- **理由**: PC-98 の32ビット環境では、I/O ウェイトを待つよりも CPU で直接 VRAM を操作したほうが高速になる（逆転現象）。
- **方式**: メインメモリ上のバックバッファに描画し、`gfx_present()` で VRAM に一括転送する。

### DMA 64KB 境界

- FDD 等の DMA バッファは 64KB 境界をまたいではならない。
- `disk.c` では BSS 配置により回避している。

## §4. ビルドとデプロイ

### 基本ルール

```bash
/* 通常ビルド */
make all

/* クリーンビルド (KernelAPI構造体変更時は必須) */
make clean
make all
```

> ⚠️ **重要**: `os32_kapi_shared.h` / `os32_kapi_generated.h` / `kapi.json` を変更した場合は、必ず `make clean` → `make all` を実行すること。古い `syscalls.o` が残ると ABI 不整合で `malloc` が全て ENOMEM で失敗する。

### KernelAPI 拡張手順

1. `tools/kapi.json` に関数プロトタイプ定義を追加
2. `make all` で API メタデータを自動生成
3. 必要に応じて `kapi/` にラッパーを実装
4. **`make clean` → `make all` で全プログラムを再ビルド**

### デプロイパスの整合性

`deploy.yaml` のゲストパスと `config.h` の `SYS_*` 定数は**必ず一致**させること。

| ビルド成果物 | ゲストパス | 設定元 |
|-------------|-----------|--------|
| `programs/shell.bin` | `/shell` | `config.h: SYS_SHELL_BIN` |
| `kernel.bin` | (NHDブート領域) | `deploy.yaml: boot.kernel` |
| `unicode.bin` | `/unicode.bin` | `config.h: SYS_UNICODE_BIN` |

不一致はサイレントに古いバイナリが使われ続ける致命的な問題を引き起こす。

### コマンド実行ルール (WSL環境)

- **`&&` でのコマンド連結は禁止**。必ず1コマンドずつ逐次実行すること。
- WSL から `/mnt/c/` (Windows側ファイルシステム) へのアクセスは遅延が大きいため、`&&` で連結するとタイムアウトの原因になる。

## §5. Git コミットポリシー

> 本セクションは旧 [GIT_POLICY.md](GIT_POLICY.md) の内容を統合したものです。

### マイクロコミットの徹底

OS・低レイヤー開発においては、後からの `git bisect` や原因調査が容易であることが命綱となる。

- **禁止事項**: AI と一緒に大量のコードをリファクタリングして、テストや複数機能が混ざったまま「一括コミット」すること。
- **推奨単位**: 「KAPIの関数を1つ追加した」「ページングの不具合を1つ修正した」「1つのファイルをリファクタリングした」という論理的な最小単位で必ず `git commit` を行う。
- **背景**: PC-98 特有の挙動不良（画面のドット異常、謎のハングアップなど）が発生した際に、どの変更で壊れたのかを追跡できるようにするため。

### Conventional Commits

コミットメッセージは以下のプレフィックスを使用する:

| プレフィックス | 用途 |
|--------------|------|
| `feat:` | 新機能の追加 |
| `fix:` | バグ修正 |
| `refactor:` | 動作を変えずにコードの構造を改善 |
| `docs:` | ドキュメントの修正 |
| `test:` | 追加テスト |

**メッセージ例:**
```
refactor(exec): 外部プログラムローダーのハードコードを排除

VFSの仕様に基づき、固定のバッファサイズ(0x4000など)を利用していた
一時的な実装を削除し、動的割り当てとOS32_MAX_PATHを利用するように修正。
```

### 多段レビュー (ベアメタル特有)

ベアメタル OS であるため、CI による自動化されたユニットテストが十分に行えない。

- **AI と人間の役割**: 動作確認は人間の目でエミュレータ (NP21/W) を起動して確認する。ビルドが通ったからといって 100% 安全だと過信せず、AI の生成コードの中に「ハードウェア制約 (EGC 使用禁止、BDA の上書きなど) に違反していないか」を人間がレビューする時間を設ける。

## §6. テストと品質保証

### 基本的な検証フロー

1. `make all` でビルドエラーがないことを確認
2. NHD にデプロイ (`nhd_deploy.py sync` → `nhd_deploy.py deploy`)
3. NP21/W を再起動
4. `ver` コマンドでビルドタイムスタンプを確認
5. 変更対象の機能を手動テスト

自動化ワークフロー (`/build-os32`, `/full-build`) を活用すること。

### シリアルコンソール経由のリモートテスト

- `os32_server.py` (HTTP API) を通じてコマンドを送信し、結果を自動取得できる。
- `curl -X POST http://localhost:8032/cmd -d "コマンド"` で実行可能。
- スクリーンショットの自動取得: `curl -sX GET http://localhost:8032/screenshot`

### テスト対象の優先度

| 優先度 | 対象 | 理由 |
|--------|------|------|
| 最高 | ブートシーケンス | 失敗するとシステム全体が起動しない |
| 高 | KernelAPI 変更 | ABI 不整合が全プログラムに波及する |
| 高 | ファイルシステム | データ破壊のリスクがある |
| 中 | シェルコマンド | ユーザー操作に直結する |
| 低 | ドキュメント変更 | 動作への影響なし |

## §7. リリースとバージョン管理

### バージョン番号の管理

| 定数 | 定義場所 | 用途 |
|------|---------|------|
| `SYS_VERSION` | `include/config.h` | OS全体のバージョン文字列 |
| `KAPI_VERSION` | `include/os32_kapi_shared.h` | KernelAPI の互換性バージョン |

### タグ付け方針

- リリース時に `v1.0`, `v1.1` 等の Git タグを付与する。
- タグ付け前に `make clean` → `make all` でフルビルドが通ることを確認する。
