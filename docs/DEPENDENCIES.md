# OS32 ソースコード依存関係定義

OS32の各モジュール間における依存関係と、設計上のレイヤー構造を以下に定義する。

## 1. レイヤー構造と垂直依存性

上位レイヤーは下位レイヤーの機能を利用するが、下位レイヤーが上位レイヤーの具体的な実装に依存すること（逆参照）は原則として禁止する。

```text
[ アプリケーション層 ] (programs/)
        |
        v
[ API・システムコール層 ] (kapi/) <--- (kapi.json から自動生成)
        |
        +-----------------------+-----------------------+
        |                       |                       |
[ ファイルシステム層 ] (fs/)  [ グラフィックス層 ] (gfx/)  [ 実行制御 ] (exec/)
        |                       |                       |
        +-----------+-----------+-----------+-----------+
                    |           |           |
              [ カーネルコア層 ] (kernel/)
                    |           |
              [ デバイスドライバ層 ] (drivers/)
                    |           |
              [ 共通定義・I/O層 ] (include/, lib/)
```

---

## 2. モジュール別詳細依存関係

### 2.1 共通基盤 (include/, lib/)
*   **依存先:** なし
*   **役割:** 基本型定義 (`types.h`)、I/Oポート操作マクロ (`io.h`)、定数定義。
*   **注意:** ここに定義されたヘッダは、カーネル・プログラムを問わずプロジェクト全域で参照される。

### 2.2 デバイスドライバ (drivers/)
*   **依存先:** `include/`, `kernel/` (kprintf/kmalloc)
*   **役割:** ハードウェア（KBD, FDC, IDE, RTC, Serial, KCG）の直接制御。
*   **特記事項:** `kcg.c` は `gfx/` の描画機能を利用せず、自前でピクセル描画（またはバックバッファ操作）を行う。

### 2.3 カーネルコア (kernel/)
*   **依存先:** `include/`, `drivers/` (serial, console用)
*   **役割:**
    *   `idt.c`, `isr_handlers.c`: 中断・例外処理。
    *   `paging.c`: メモリ保護と仮想アドレス管理。
    *   `kmalloc.c`: カーネル内メモリアロケータ。
    *   `console.c`: テキスト表示とログ出力。

### 2.4 ファイルシステム (fs/)
*   **依存先:** `include/`, `drivers/disk.h`, `kernel/` (kmalloc)
*   **役割:** VFS（仮想ファイルシステム）による抽象化と、FAT12/ext2/serialfsの実装。
*   **注意:** `vfs.c` は全てのファイルシステム実装の親となり、`drivers/disk.c` を通じて物理デバイスにアクセスする。

### 2.5 グラフィックス (gfx/)
*   **依存先:** `include/`, `drivers/kcg.h` (フォント), `kernel/` (dirty領域追跡)
*   **役割:** バックバッファ管理、VRAM転送、スプライト描画、図形描画。
*   **最適化:** `gfx_sprite.c` は高速化のため `gfx/gfx_asm.asm` (アセンブラ) に依存する。

### 2.6 プログラムローダー (exec/)
*   **依存先:** `fs/`, `kernel/paging.h`, `kapi/`
*   **役割:** 外部実行ファイル（.EXE/OS32X）のロード、メモリ展開、実行。
*   **注意:** アプリケーションのヒープ管理 (`exec_heap.c`) も担当する。

### 2.7 KernelAPI (kapi/)
*   **依存先:** `kernel/`, `fs/`, `gfx/`, `exec/`, `drivers/`
*   **役割:** カーネル機能を外部プログラムに公開するためのゲートウェイ。
*   **生成プロセス:** `tools/kapi.json` → `kapi_generated.c` / `os32_kapi_generated.h`。

## 3. ヘッダファイル参照ルール

### 3.1 参照パス
Makefileの設定により、以下のディレクトリは `-I` 指定されているため、ディレクトリ名を省略して `#include "file.h"` と記述できる。
*   `include/`
*   `kernel/`
*   `drivers/`
*   `gfx/`
*   `fs/`

### 3.2 循環参照の回避
*   `A.h` が `B.h` を含み、`B.h` が `A.h` を含むことは禁止する。
*   構造体の参照が必要な場合は、前方宣言 (`struct MyStruct;`) を活用し、可能な限り `.c` ファイル内でインクルードする。

## 4. ビルド時の依存関係

1.  **KAPI生成:** `kapi.json` から `kapi_generated.c` を生成。
2.  **カーネルビルド:** 全てのカーネルオブジェクトをリンクして `kernel.bin` を作成。
3.  **プログラムビルド:** `libos32` (KAPIラッパー) とリンクして各アプリを作成。
4.  **イメージ作成:** `mkfat12.py` 等でディスクイメージに配置。
