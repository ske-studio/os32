# DEBUG.md - OS32 メモリ管理の依存関係と調査方針

公式ドキュメント（第2部 メモリマップ、第9部 exec）の更新内容を反映し、現在の「プロセスごとに仮想メモリが完全に分離・独立した環境 (VMM)」に基づくメモリ管理の依存関係と、システムが抱えていると疑われる問題点（メモリリーク・不正アクセス・パニック等）の調査方針を再度整理したドキュメントです。

## 1. メモリ管理サブシステムの全体像と依存関係

現在、かつての「子プロセス領域」などは廃止され、全プロセスがアイデンティティマッピングから解放されて完全に独立したメモリ空間（ベースアドレス 0x400000）を動的に与えられています。

1. **物理メモリ管理 (PMM)** - `kernel/pmm.c`
   * **役割**: 4KBページ単位での物理メモリ割り当て/解放（ビットマップ方式）。
   * **依存先**: なし
2. **仮想メモリ管理 (VMM / Paging)** - `kernel/paging.c`
   * **役割**: プロセスごとの独立したページディレクトリ（PDI=1に 0x400000〜 のみをマッピング）の生成・破棄。
   * **依存先**: PMM
3. **プロセス実行・構築 (Exec Engine)** - `exec/exec.c`
   * **役割**: `vfs_read()`による対象バイナリのロード、VMMによるプロセス空間の切り替え(`CR3`), 指定サイズ分のスタック・ヒープの確保とセットアップ。
   * **依存先**: PMM, VMM, カーネルヒープ (kmalloc)
4. **プロセス用ヒープ管理 (Exec Heap)** - `exec/exec_heap.c` / `syscalls.c`
   * **役割**: 分離された仮想メモリ空間内（`pd[1]`）での `malloc` / `_sbrk` 処理。
   * **依存先**: Exec Engine によって確保されたヒープ開始アドレス。

## 2. 問題箇所調査のために確認すべき重点項目

完全仮想化への移行に伴い、**「仮想アドレスのコンテキスト（CR3）不一致によるメモリ破壊・不正読み込み」**のリスクが高まっています。以下の点をロジックから重点的に検証する必要があります。

### A. プロセス切り替え時のポインタまたぎ（CR3不一致による不正メモリアクセス）★最重要★
* **対象**: `exec/exec.c`内の `exec_run()` と `exec_setup_stack()`
* **仮説**: 
  親プログラム（例: shell）がカーネル関数(`exec_run`)を呼び出した際、引数の `cmdline` ポインタは **「親プログラムの仮想空間（親の 0x400000〜）」** を指している。
* **検証ポイント**:
  1. ロードのために `paging_switch_dir(new_pd)` で子プロセスの空間に切り替えた**後**に、親プロセス側のポインタである `cmdline` を辿って引数文字列を読み取ろうとしていないか？
  2. 切り替え後の空間には既に「子プロセスのバイナリ」が上書きされており、`cmdline` ポインタが指す先は親の文字列ではなく、子のコードや未初期化メモリのガベージデータの読み込みになる。これが異常終了やパニックの原因になっている可能性が高い。

### B. 例外（Fault / Exit）時のページディレクトリ解放の整合性
* **対象**: `exec/exec.c`内の `error_exit` および `exec_longjmp` のリカバリ
* **検証ポイント**: 
  確実に `old_pd` へコンテキストを戻してから、不要になった `new_pd` およびその物理ページ(`pmm_free_page`)を解放しているか。

### C. ページ境界（ページサイズ）のアライメント違反
* **対象**: プログラムのメモリ計算とPMMのアロケーション
* **検証ポイント**:
  * Execエンジンにおける `total_prog_pages` やスタックページの計算で、ヒープサイズやBSSの端数が考慮から漏れて境界オーバーを起こしていないか。
  * スタックトップ（`0x57F000`）から下方への物理ページマッピングで境界ズレが生じていないか。
