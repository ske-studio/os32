# OS32開発におけるAIと協調するGitポリシー

AIコーディングアシスタント（Cursor, Copilot, Gemini等）を活用してOS32を開発するにあたり、「開発スピードを落とさずに、いかにシステムとしての品質・履歴の可読性を担保するか」を目的とした公式ガイドラインです。

## 1. マイクロコミット（小規模コミット）の徹底
OS・低レイヤー開発においては、後からの `git bisect` や原因調査が容易であることが命綱となります。

* **禁止事項:** AIと一緒に大量のコードをリファクタリングして、テストや複数機能が混ざったまま「一括コミット」することは避けてください。
* **推奨単位:** 「KAPIの関数を1つ追加した」「ページングの不具合を1つ修正した」「1つのファイルをリファクタリングした」という論理的な最小単位で必ず `git commit` を行います。
* **背景:** 万が一、PC-98特有の挙動不良（画面のドット異常、謎のハングアップなど）が発生した際に、どの変更で壊れたのかを追跡できるようにするためです。

## 2. コミットメッセージの明確化（Conventional Commits）
Gitの履歴は「変更点」ではなく「変更の意図」を残す場所です。コミットメッセージは以下のプレフィックスを使い、AIを利用して効率的に作成してください。

* `feat:`: 新機能の追加
* `fix:`: バグ修正
* `refactor:`: 動作を変えずにコードの構造を改善
* `docs:`: ドキュメントの修正
* `test:`: 追加テスト

**メッセージ例:**
```text
refactor(exec): 外部プログラムローダーのハードコードを排除

VFSの仕様に基づき、固定のバッファサイズ(0x4000など)を利用していた
一時的な実装を削除し、動的割り当てとOS32_MAX_PATHを利用するように修正。
```

## 3. 自動テストが難しい環境での「多段レビュー」
ベアメタルOSであるため、CIによる自動化されたユニットテストが十分に行えません。

* **AIと人間の役割:** 動作確認は、人間の目でエミュレータ（NP21/W）を起動して確認するしかありません。ビルドが通ったからといって100%安全だと過信せず、AIの生成コードの中に「ハードウェア制約（EGC使用禁止、BDAの上書きなど）に違反していないか」を人間がレビューする時間を設けてください。

## 4. コンテキストと指示書のバージョン管理（Docs as Code）
AIアシスタントに、プロジェクト独自の制約を確実に認識させることは非常に重要です。

* このリポジトリではAI向けの絶対ルールを `gemini.md` またはそれに準じる設定ファイル（`.cursorrules`等）でバージョン管理しています。
* 「定数の3層管理ポリシー」「C89（GNU89）の文法厳守」「ハードウェアアクセラレータ使用禁止」などは、暗黙のルールにせず常にこれらの指示書に明記してください。変更があった際は必ず指示書も同時に更新します。
