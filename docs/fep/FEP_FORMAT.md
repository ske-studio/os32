# OS32 FEP 辞書フォーマット仕様 v2 (*.dic)

本ドキュメントは、OS32のFEPに用いられるコンパイル済み辞書バイナリ(`.dic`)のフォーマット仕様を定義する。

## 1. 概要

- **設計思想**: 2階層インデックス（Level-1 + Level-2）を起動時にオンメモリ常駐させ、変換時は対象ブロックのみをディスクからロードする。
- **エンディアン**: リトルエンディアン
- **文字コード**: UTF-8
- **常駐メモリ**: 約44KB (ヘッダ + Level-1 + Level-2 インデックス)

### 構造概要

```
[Header]           16 bytes
[Level-1 Index]    L1_count × 12 bytes
[Level-2 Index]    L2_count × 12 bytes
[Data Blocks]      可変長 (WordEntry の連続)
```

## 2. ヘッダ (16 バイト)

| オフセット | サイズ | 名称 | 内容 |
| :--- | :--- | :--- | :--- |
| 0x00 | 4 | `magic` | `0x54434944` ("DICT" リトルエンディアン) |
| 0x04 | 4 | `version` | 辞書バージョン (現在: `2`) |
| 0x08 | 4 | `total_words` | 辞書の総単語数 |
| 0x0C | 4 | `l1_count` | Level-1 インデックスのエントリ数 |

## 3. Level-1 インデックス (エントリあたり 12 バイト)

読みの **先頭1文字** をキーとする。各エントリは、対応する Level-2 サブインデックスの位置と個数を示す。

| オフセット | サイズ | 名称 | 内容 |
| :--- | :--- | :--- | :--- |
| +0 | 4 | `key_char` | 読みの先頭1文字 (UTF-8, パディングは `0x00`) |
| +4 | 4 | `l2_offset` | この文字に属する Level-2 エントリの開始位置 (ファイル先頭からの絶対オフセット) |
| +8 | 4 | `l2_count` | この文字に属する Level-2 エントリ数 |

## 4. Level-2 インデックス (エントリあたり 12 バイト)

読みの **先頭2文字** をキーとする。各エントリは、対応するデータブロックの開始位置を示す。
1文字しかない読みの場合は、1文字がキーとなる。

| オフセット | サイズ | 名称 | 内容 |
| :--- | :--- | :--- | :--- |
| +0 | 8 | `key_chars` | 読みの先頭2文字 (UTF-8, パディングは `0x00`) |
| +4 | 4 | `data_offset` | データブロックの開始位置 (ファイル先頭からの絶対オフセット) |

## 5. データブロック (可変長)

各 Level-2 キーに対応するデータブロックには、そのキーで始まる全ての単語が連続して格納される。
各単語は以下の `Word Entry` 構造を持つ。

### Word Entry 構造

| サイズ | 名称 | 内容 |
| :--- | :--- | :--- |
| 4 | `WordMeta32` | メタデータパック構造体 |
| 可変長 | `yomi` | 読み仮名 (UTF-8, ヌル終端なし, 長さは `WordMeta32.yomi_len`) |
| 可変長 | `kanji` | 表層形 (UTF-8, ヌル終端なし, 長さは `WordMeta32.kanji_len`) |

### WordMeta32 ビットレイアウト (32bit)

| ビット | 幅 | 名称 | 内容 |
| :--- | :--- | :--- | :--- |
| 0-4 | 5 | `yomi_len` | `yomi` のバイト長 (0〜31) |
| 5-9 | 5 | `kanji_len` | `kanji` のバイト長 (0〜31) |
| 10-20 | 11 | `pos_id` | 品詞/文脈ID (0〜2047) |
| 21-31 | 11 | `cost` | 出現コスト (0〜2047, 小さいほど優先) |

## 6. 実機側の読み込みシーケンス

### 起動時 (常駐ロード)

1. `.dic` ファイルをオープンし、ヘッダ 16B を読み込む。
2. `magic` と `version` を検証。
3. Level-1 インデックス (`l1_count × 12B`) をメモリに読み込む。
4. Level-2 インデックス全体をメモリに読み込む。
   - Level-2 の総エントリ数は、全 Level-1 エントリの `l2_count` を合計して算出する。
   - または、`(Level-2開始位置 〜 最初のデータブロック開始位置)` の範囲から算出可能。
5. ファイルをクローズ (変換時に再オープン)。
6. **常駐メモリ: 約44KB**

### 変換時 (ディスクI/O)

ユーザーが「**かんじ**」と入力して変換キーを押した場合:

1. 先頭1文字「**か**」で Level-1 を検索 → Level-2 のオフセットとエントリ数を取得。
2. 先頭2文字「**かん**」で Level-2 を二分探索 → データブロックのオフセットを取得。
3. データブロックのサイズを算出 (次の Level-2 エントリの `data_offset` との差)。
4. `.dic` をオープンし、`lseek` + `read` でデータブロックをロード。
5. ブロック内をシーケンシャルにパースし、`yomi` が「**かんじ**」に前方一致するエントリを抽出。
6. `cost` 順でソートして候補を返す。

## 7. ブロックキャッシュ (実機側)

変換時のディスクI/Oを削減するため、最近ロードしたデータブロックを固定サイズのキャッシュバッファに保持する。

### 仕様

| 項目 | 値 |
|:---|:---|
| キャッシュバッファサイズ | **16,384 bytes (16KB)** 固定 |
| 確保タイミング | FEP初期化時に `mem_alloc(16384)` で1回 |
| 淘汰方式 | FIFO (先頭の最古エントリから破棄) |
| 16KBを超えるブロック | **キャッシュしない** (一時バッファに直接ロードして使い捨て) |

### キャッシュエントリ構造 (エントリごとにヘッダ12B + データ)

```
[CacheEntry Header]  12 bytes
  +0: l2_key       (8B)  Level-2 キー (照合用)
  +8: block_size   (4B)  データブロックのサイズ
[Block Data]         block_size bytes
  (WordEntry の連続データ、そのまま格納)
```

### 動作フロー

1. 変換時、Level-2 キーでキャッシュ内を線形検索する。
2. **ヒット**: ディスクI/Oをスキップし、キャッシュ上のデータを直接パースする。
3. **ミス**: ディスクからデータブロックを読み込み、キャッシュに追加する。
   - 空き容量が足りない場合、先頭(最古)のエントリから順に破棄して空きを確保する。
   - ブロックサイズ + 12B (ヘッダ) > 16KB の場合はキャッシュに格納しない。

### 期待される効果

中央値181Bのブロックであれば約80件をキャッシュ可能。同じ先頭2文字の変換を繰り返す場合（例：「しょうがくせい」「しょうにん」など）は2回目以降ディスクI/Oがゼロになる。

### キャッシュの永続化 (`fep.cache`)

キャッシュバッファの16KBをそのまま `fep.cache` ファイルとしてディスクに書き出し・復元することで、次回起動時に即座にウォームキャッシュの状態でFEPを利用できるようにする。

| 項目 | 内容 |
|:---|:---|
| ファイル名 | `fep.cache` (辞書と同じディレクトリ) |
| ファイルサイズ | 常に16,384 bytes (バッファのメモリイメージそのもの) |
| 保存タイミング | `sys_reboot` / シャットダウン処理時 |
| 復元タイミング | FEP初期化時。ファイルが存在すれば `sys_read` で一括ロード |
| ファイル不在時 | キャッシュをゼロクリアして空の状態で開始（コールドスタート） |
| 破損検出 | ファイルサイズ ≠ 16,384 の場合は無視してコールドスタート |
