# FEP辞書コンパイラ：Pythonライブラリ調査および技術選定

本ドキュメントでは、IPADIC(CSV)からOS32用独自フォーマット(`.dic`)を構築するためのPythonライブラリの調査結果と、最終的な技術選定のアプローチをまとめます。

## 1. CSVの読み込みとパース

**要件**: IPADIC等の数万〜数十万行のCSVを高速に読み込み、特定の列（単語、読み、品詞番号、コスト値等）を抽出する。文字コードは **EUC-JP** である（ダウンロード実績により確認済み）。

| ライブラリ | 種別 | 特徴・評価 |
| :--- | :--- | :--- |
| `csv` | 標準 | Pythonの組み込みモジュール。数万行なら1〜2秒程度で処理可能。追加の依存関係を生まないのが最大のメリット。 |
| `pandas` | 外部 | C実装ベースで非常に高速。ただしインストールサイズが大きく、ツールを使う他の開発者の環境構築ハードルが上がる。 |

**結論・提案**:
**`pandas`** を採用します。大量のCSV（約24万行）のフィルタリング・ソート・集約処理に適しており、`pd.read_csv(file, encoding='euc-jp')` で直接読み込み可能です。

## 2. 文字エンコーディング・表示可能性チェック

**要件**: 抽出した単語が、PC-98の実機（JIS / シフトJIS / PC-98拡張文字）で表示・印字可能であるか判定し、表示できない文字が含まれる単語を除外する。

| ライブラリ | 種別 | 特徴・評価 |
| :--- | :--- | :--- |
| `codecs` (組込 `encode`) | 標準 | Pythonの `string.encode('cp932', errors='strict')` 等を用いることで、変換出来ない文字があった場合に例外(`UnicodeEncodeError`)をキャッチして除外可能。 |
| `nkf` モジュール | 外部 | 変換専用の高機能モジュールだが、大掛かりすぎる。 |

**結論・提案**:
**標準機能 `.encode('cp932')`** の例外機構を利用して判定します。

**補足（実機制約）**: OS32カーネルの `console.c` は、Unicode→JISコード変換テーブル (`unicode_to_jis()`) を内蔵しており、JIS第1水準・第2水準の文字のみが表示可能です。CP932に変換可能であっても、JIS第1/第2水準の範囲外の文字（NEC特殊文字、IBM拡張文字等）は辞書に含めない方が安全です。より厳密なフィルタリングを行う場合は、OS32側の `unicode.bin` テーブルとの整合性検証も検討対象です。

## 3. 日本語（読み仮名）のソート

**要件**: 読み仮名に基づき、辞書順（あいうえお順）にデータを並び替える。

| ライブラリ | 種別 | 特徴・評価 |
| :--- | :--- | :--- |
| 標準 `sort()` | 標準 | IPADICの「読み」は全て全角カタカナで統一されているため、標準のPython文字列比較（Unicodeのコードポイント比較）で実用上問題のない五十音順になる。 |
| `jaconv` | 外部 | ひらがな/カタカナ変換などを提供するが、入力データがカタカナに統一されているなら不要。 |

**結論・提案**:
外部ライブラリは不要。**標準のリストメソッド `.sort(key=...)`** を用います。

**補足**: FEP側ではユーザーがひらがなで入力します。検索時にひらがな→カタカナ変換が必要ですが、これは `ord(c) + 0x60` のような単純なコードポイントシフトで実現できるため、外部ライブラリは不要です。

## 4. バイナリパック（32bit構造体生成）

**要件**: 複数のメタデータ（例: yomi_len(5), kanji_len(5), cost(12), pos_id(10)）を高速・確実に4バイト（uint32_t リトルエンディアン）のバイナリデータにパックする。

| ライブラリ | 種別 | 特徴・評価 |
| :--- | :--- | :--- |
| **ビット演算** + `struct` | 標準 | 各値をビットシフト(`<<`)と論理和(`\|`)で1つの32bit整数に結合し、`struct.pack('<I', value)` でリトルエンディアンバイナリ化する。コードは泥臭いが、エンディアンの差異が出ず最も確実で高速。 |
| `ctypes` (Structure) | 標準 | C言語のビットフィールドをPython上で定義できる。記述はエレガントになるが、ホストPCのエンディアン等の環境依存を受けやすく、パディングの制御に癖がある。 |
| `bitstruct` / `construct` | 外部 | 任意ビット数のバイナリパックに特化したライブラリだが、外部依存が増加する。 |

**結論・提案**:
最も堅牢で依存のない **「Python標準のビット演算を利用して整数化 ＋ `struct.pack` によるバイナリ化」** を採用します。

## 5. IPADIC CSVフォーマット仕様（調査結果）

ダウンロードした辞書データ（EUC-JP）のカラム構成を調査した結果は以下の通りです。

```
表層形, 左文脈ID, 右文脈ID, コスト, 品詞, 品詞細分類1, 品詞細分類2, 品詞細分類3, 活用型, 活用形, 原形, 読み, 発音
```

| カラム番号 | 内容 | 例(Noun.csv) | 例(Verb.csv) | FEPでの利用 |
| :---: | :--- | :--- | :--- | :--- |
| 0 | 表層形 | `仕舞い` | `引き込む` | 変換結果の表記 |
| 1 | 左文脈ID | `1285` | `762` | `pos_id`として利用可能 |
| 2 | 右文脈ID | `1285` | `762` | （不使用） |
| 3 | コスト | `5543` | `7122` | `cost`としてスケーリング圧縮 |
| 4 | 品詞 | `名詞` | `動詞` | フィルタリング用 |
| 5 | 品詞細分類1 | `一般` | `自立` | フィルタリング用 |
| 6-9 | 品詞細分類2〜活用形 | `*,*,*,*` | `*,*,五段・マ行,基本形` | 活用処理に利用（Phase2以降で検討） |
| 10 | 原形 | `仕舞い` | `引き込む` | （表層形と同一の場合が多い） |
| 11 | 読み | `シマイ` | `ヒキコム` | 検索キー（全角カタカナ） |
| 12 | 発音 | `シマイ` | `ヒキコム` | （不使用） |

### 動詞・形容詞の活用形問題

**重要**: `Verb.csv` と `Adj.csv` にはカラム9の「活用形」ごとに表層形が異なる行が存在します（例: 「引き込む」「引き込ま」「引き込み」…は同一動詞の別活用形）。

Phase 1段階では、全エントリを素直に読み込んで変換候補とするが、初期版のFEPは連文節変換を行わないため、**辞書サイズ削減のため活用形「基本形」のみを抽出する**フィルタリングが有効です。

## 総合的な提案アプローチ

本フェーズの実装では **外部のサードパーティライブラリ（pip install が必要なもの）は一切使用せず、Pythonの標準ライブラリのみで完結させる** 構成を提案します。
これにより、自作OSプロジェクトのビルドチェーンに組み込んだ際のポータビリティが最大化されます。
