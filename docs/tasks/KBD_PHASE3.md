# Phase 3: キーボードタイプ識別 (0x9Fh) — 実機互換性

## 概要
UNDOCUMENTED io_kb.md の下りコマンド `0x9Fh` を使用し、接続されているキーボードの種類を自動判別する。実機で異なるキーボードタイプに応じたキーマップの自動選択を可能にする。

## 背景
- PC-98にはPC9800Bible記載だけでも複数種類のキーボードが存在する
- UNDOCUMENTEDでは旧キーボード4種 + 新キーボード8種の分類を詳述
- NP21/Wでは常に新キーボード相当だが、実機ではキーボードによりキーコードが異なる

## 出典
- UNDOCUMENTED 9801/9821 Vol.2 io_kb.md — I/O 0041h WRITE コマンド `9Fh`

## 9Fhコマンド仕様
```
送信: 9Fh → キーボードからタイプ識別コードを返答
  旧キーボード: 応答なし（タイムアウトで判別）
  新キーボード: 識別コードを返す
    - PC-9801-98:  A0h返答後にサブタイプ
    - PC-9801-106: A0h返答後にサブタイプ
    - PC-9801-119: 同上（Windowsキーあり）
    等
```

## タスクリスト
- [ ] `kbd_identify()` 関数を実装
  - `kbd_send_cmd(0x9F)` 送信後、一定時間内に返答がなければ「旧キーボード」と判定
  - 返答があればキーボードタイプを解析
- [ ] キーボードタイプ列挙定数を定義
  ```
  KBD_TYPE_OLD      旧キーボード（応答なし）
  KBD_TYPE_98       PC-9801-98
  KBD_TYPE_106      PC-9801-106
  KBD_TYPE_109      PC-9801-109 (NX互換)
  KBD_TYPE_119      PC-9801-119 (Windowsキー付き)
  KBD_TYPE_UNKNOWN  その他
  ```
- [ ] `kernel_main()` 初期化時に `kbd_identify()` を呼び、結果をグローバル変数に保存
- [ ] キーマップテーブルをタイプ別に分岐する基盤を設計（タスク外で実装）
- [ ] `ver` コマンドなどでキーボードタイプを表示

## 依存関係
- Phase 1 (kbd_send_cmd 基盤) が先に必要
- キーボードからの「返答」受信機構 (Phase 2 と共通)

## 注意事項
- 旧キーボードは下りコマンドを一切認識しない。タイムアウトで「旧」と判定する必要がある
- NP21/Wでの `0x9Fh` 応答の有無は要確認
- 実機が手元にない場合、テスト困難な領域。NP21/Wの挙動のみで検証する場合のフォールバック設計が重要
