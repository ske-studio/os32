# FEP Phase 4: FEP UI（画面描画）との統合

## 目的
テキストベースの独自画面制御を利用して、ユーザーのアプリケーション画面上に変換コンテキストや候補ウィンドウを非破壊的に描画し、実用的な日本語入力のユーザー体験（UX）を提供する。

## 前提条件
- Phase 2 の検索エンジンと Phase 3 のステートマシンが動作済み
- 既存の TVRAM 操作 API:
  - `tvram_clear`, `tvram_putchar_at(x, y, ch, attr)`, `tvram_putkanji_at(x, y, jis, attr)`, `tvram_scroll`
  - `console_set_cursor(x, y)`, `console_get_cursor_x/y`, `console_get_size(w, h)`
  - `shell_print_utf8(str, color)` はカーソル追従付きUTF-8表示（`unicode_to_jis()` で漢字ROM利用）
- テキストVRAMレイアウト (PC-98):
  - テキストVRAM: `0xA0000`〜 (80桁×25行, 1文字=2B文字コード + 2B属性)
  - 属性VRAM: `0xA2000`〜
  - 文字は80桁 × 25行（行0〜24）、行24が25行目
- グラフィックVRAM: `0xA8000`〜 (4プレーン × 32KB)

## タスクリスト

### Phase 4a: VRAM描画基盤の対応
- [ ] 25行目（行24）のテキストVRAMへの直接書き込みによるFEP表示領域ロジックの実装
  - 未確定文字列（ひらがな入力中テキスト）の表示
  - 変換候補の表示（cost順のリスト）
  - ステータスインジケーター（[あ] / [A] など入力モード表示）
- [ ] アプリケーションがグラフィックスモードを利用中の場合のVRAM退避(save)・復元(restore)機構の実装
  - グラフィックVRAMの下部（25行目相当領域: Y=384〜399, 16ライン分）を4プレーン分退避
  - 退避先: `mem_alloc` で確保した一時バッファ（80B × 16lines × 4planes = 5,120B）
  - FEP表示終了時に退避データを書き戻し
- [ ] 漢字ROM (`kcg_read_kanji`) を利用した候補文字列の高速描画ロジック
  - テキストVRAMの漢字表示は `tvram_putkanji_at` を利用（既存API）
  - グラフィックモード時はフレームバッファプレーンへの直接ビットマップ描画
- [ ] テキストVRAM 1行目〜24行目（行0〜23）の表示・非表示切り替え機能
  - GDCのテキスト表示プレーン制御（テキストVRAMの表示ON/OFF）の実装
  - PC-98 GDC: `0xA2` (テキストGDCコマンドポート) 、表示開始アドレスの変更、またはプレーンマスク制御
  - この機能は「FEP起動時のみ25行目を強調表示する」等のモードに利用
  - API設計: `tvram_set_visible(int start_row, int end_row, int visible)` のような関数

### Phase 4b: 画面制御とUIフローの統合
- [ ] Google日本語入力仕様に則ったUIインタラクションの実装
  - **未確定文字列の表示**: 25行目左端にひらがなの入力中テキストを描画。属性(色)を通常テキストと区別
  - **変換候補ウィンドウ**: 候補リスト（最大N件表示）を25行目の右側、または上部領域にオーバーレイ表示
  - **候補選択**: スペースで次候補、↑/↓でリスト送り、数字キー(1-9)によるダイレクト選択
  - **候補ページング**: 候補が多い場合のページ送り表示（Tab / Shift+Tab等）
- [ ] Enterによる「確定」時のUIクリアとVRAM復元処理
- [ ] Escapeによるキャンセル（未確定テキスト全消去、元の画面に復帰）

### Phase 4c: 最終検証と常駐化（最終ステップ）
- [ ] テストアプリケーションでの全通し動作確認
  - テキストモードアプリ（shell, vz等）でのFEP動作検証
  - グラフィックモードアプリ（gfx_demo等）でのFEP動作検証（VRAM退避・復元）
- [ ] このプロセス・モジュールをカーネルまたはOS起動時の常駐プロセスとして適切に組み込む最終対応（アーキテクチャ統合）
  - `kbd_irq_handler` からのコールバックチェーン導入の検討
  - FEP関連KernelAPI（`fep_enable`, `fep_disable`, `fep_get_mode` 等）の追加
- [ ] 今後の文字コード対応（カーネル側のShift-JISサポートなど）に向けた課題整理
- [ ] 辞書ファイルの最終配置パスの決定とドキュメント化

## 成果物
- VRAM退避・復旧を含む画面制御・描画モジュール (`lib/fep_ui.c`)
- テキストVRAM 表示制御用関数（`tvram_set_visible` 等）
- 完全な動作をするFEPテストアプリケーション
- 常駐化されたFEPカーネルモジュール

## 設計上の注意点
- **テキストVRAM 25行目の衝突**: 現在のコンソール (`console.c`) は `TVRAM_ROWS` (25行) を使用しており、スクロール時に25行目も巻き込む。FEP表示領域を保護するために、`tvram_scroll` を24行分のスクロールに制限する変更が必要。
- **GFX/テキスト混在**: グラフィックモード中にテキストVRAMを表示する場合、GDCのテキスト/グラフィック両方の表示を有効にする必要がある。PC-98では `0x68` ポート（画面モード制御）で制御可能。
- **漢字ROM利用制約**: 漢字ROM (`kcg_read_kanji`) はJIS第1/第2水準のグリフのみ提供。第3/第4水準の文字が変換候補に含まれないようにする辞書コンパイル段階（Phase 1）でのフィルタリングが前提条件。
