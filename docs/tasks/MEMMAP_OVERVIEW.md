# メモリマップ再構築 — 全体概要

## 目的

OS32の物理メモリマップを整理し、以下の問題を解決する。

1. **sbrk_heap_limit バグ**: クリップ前の値で設定されsbrkがスタック/exec_heapを侵食する
2. **デッドゾーン**: 合計 ~8MB の未使用メモリ散在
3. **保護の不在**: プログラムスタックとsbrkヒープ間にガードページなし
4. **ゴースト領域**: `MEM_GFX_SURF_POOL` が完全未使用
5. **コメント乖離**: paging.c の保護マップコメントが現実と不一致

## フェーズ構成

| フェーズ | 内容 | リスク | 依存 |
|---------|------|--------|------|
| **Phase 1** | memmap.h 定数再定義 (コード変更なし) | 低 | なし |
| **Phase 2** | コンベンショナルメモリ (KHEAP拡張 + paging.cコメント修正) | 低 | Phase 1 |
| **Phase 3** | カーネルデータ再配置 (フォント/Unicode/BB/KAPI移動) | 中 | Phase 2 |
| **Phase 4** | プログラム空間再設計 (動的exec_heap + sbrk修正 + ガードページ) | 高 | Phase 3 |
| **Phase 5** | 検証 + ドキュメント更新 | 低 | Phase 4 |

## 新メモリマップ全体図

```
  0x00000 ┌───────────────────────────┐
          │  コンベンショナルメモリ      │ Phase 2: KHEAP 拡張
  0xFFFFF ├───────────────────────────┤
  0x100000│  カーネルデータ (固定)       │ Phase 3: 再配置
  0x1FFFFF├───────────────────────────┤
  0x200000│  カーネル予約 (将来拡張)    │ Phase 3: NOT PRESENT設定
  0x3FFFFF├═══════════════════════════┤ ← PDE境界 (4MB)
  0x400000│  プログラムコード + sbrk    │ Phase 4: 動的レイアウト
          │  ★ GUARD A (sbrk上限)      │
          │  exec_heap (自動拡大)       │
          │  ★ GUARD B (スタック保護)   │
          │  プログラムスタック          │
  mem_end ├───────────────────────────┤ ← 物理メモリ上限
          │  NOT PRESENT               │
  0xFFFFFF└───────────────────────────┘ ← ページング上限 (16MB)
```
