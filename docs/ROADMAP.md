## 1. v1.x GUIシェル — 「デスクトップ革命」

**コンセプト**: Windows 3.1 / 早期 Windows 95 をモデルとしたグラフィカルデスクトップ環境。OS32のCLI・グラフィックス・ファイルシステム全技術の集大成。

### 設計方針

| 項目 | 決定 |
|------|------|
| デザインモデル | **Win3.1 / 早期Win95** — シングルタスクGUIデスクトップ |
| 描画方式 | **全面GFX描画** (640×400×16色)、テキストGDC OFF |
| GUI起動方法 | CUIデフォルト + `os32gui` コマンド (コンフィグ書換え+リブート) |
| GUI API | **libos32gui** — Win32API風のGUIアプリケーション開発ライブラリ |
| メモリ配置 | シェル常駐帯域を1MBに拡張 (0x300000-0x3FFFFF) |
| GFXプログラム共存 | フルスクリーン占有 + 終了後GUI再描画 |
| CUI/GUI設定 | `/etc/system.cfg` (`GUI=0`/`GUI=1`) — FDDから復旧可能 |
| フォントキャッシュ | 既存KCGキャッシュ (295KB@0x100000) をそのまま使用 |
| GUI→CUI切替 | **明示的なGUIアクションが必須**。自動復帰なし |

### 技術基盤 (既存資産の活用)

| 既存資産 | GUIシェルでの活用 |
|---------|-----------------|
| libos32gfx | ウィンドウフレーム・アイコン・デスクトップの描画 |
| lconsole (差分仮想VRAM) | ターミナルウィンドウの中核 (80×25セル→KCGでGVRAM描画) |
| ダーティ矩形転送 | ウィンドウ単位の部分VRAM更新 |
| KCGフォントキャッシュ | ANK+漢字のグラフィカルテキスト描画 |
| スプライト機構 | マウスカーソル描画 |
| マウスI/O定義 (pc98.h) | マウスドライバの基盤 |

### CUI → GUI 切替フロー

```
[CUI → GUI]
1. CUIシェルで `os32gui` 実行
2. /etc/system.cfg に GUI=1 書込み
3. sys_reboot() で再起動
4. カーネル: system.cfg 読取 → GUI=1
5. gshell.bin を Level 0 として起動

[GUI → CUI] ※明示的なGUIアクションが必須
1. ユーザが GUI上で明示的に「CUIに切替」アクションを実行
   (スタートメニュー → 「CUIモードで再起動」等)
2. gshell が /etc/system.cfg に GUI=0 書込み
3. sys_reboot() で再起動
4. カーネル: system.cfg 読取 → GUI=0
5. shell.bin を Level 0 として起動
※ 何もしなければ次回もGUIで起動する (永続設定)

[GUI終了プロセス]
1. ユーザが「シャットダウン」を選択
2. gshell: 全ウィンドウにクローズ通知
3. gshell: system.cfg は変更しない (次回もGUI起動)
4. gshell: gfx_shutdown() + exit
5. カーネル: テキストGDC START + system.cfg 再読取
6. カーネル: GUI=1 → gshell を再起動
   (GUI=0 に変更されていた場合のみ CUI shell を起動)

[ハング復旧]
1. FDDからブート → mount /hd0
2. edit /hd0/etc/system.cfg → GUI=0 に変更
3. リブート → CUIで起動
```

#### CUI/GUI抽象化レイヤーの設計

CUIプログラム (既存42本) をGUIターミナル上でも変更なしで動作させるための抽象化:

```
[CUI環境]                      [GUI環境]
  アプリ                         アプリ
    ↓                              ↓
  KernelAPI                      KernelAPI
  (shell_print等)                (同じAPI呼び出し)
    ↓                              ↓
  console.c (TVRAM直接)          console.c → gshellのターミナルウィンドウ
```

- 既存KAPIの `shell_print` / `console_write` をGUIモードで自動的にターミナルウィンドウにリダイレクト
- `kbd_getchar` / `kbd_getkey` をGUIイベントキューから取得するように切替
- GUI専用アプリは `libos32gui` APIを直接使用し、ウィンドウ・コントロールを操作

## 2. 長期ロードマップ (v2.0+)

### 協調型マルチタスク

v1.xはシングルタスクだが、v2.0ではタイマー割り込みベースの**協調型マルチタスク**を検討:
- ウィンドウごとに独立したプロセス
- メッセージパッシングによるIPC (Win16 メッセージキュー方式)
- 子プログラム実行中もデスクトップが応答可能

### 16bit DOSプログラム移植スキーム

```
[DOS 16bit .COM/.EXE]
        ↓
  逆アセンブル / 静的解析
        ↓
  INT 21h → KernelAPI マッピング
        ↓
  32bit OS32X として再コンパイル
```

- 半自動 + 手動修正が現実的
- 小さなツール (数KBの .COM) をケーススタディとして開始
- INT 21h → KernelAPI 互換レイヤーの設計が核心
