## Part A: INT 1Bh BIOS デバッグ記録

### A-1. INT 1Bh READ (AH=06h) の BX パラメータ

#### 問題

PC-9800 Bible §2-9-4 の仕様テキストには以下のように記載されている:

```
BX = データブロック数 (n)  256×nバイト
```

この記述から `BX=2` で `256×2 = 512バイト`（1セクタ）と解釈してコードを実装した。

#### 実測結果

バッファを `0xAA` で事前充填してから INT 1Bh READ を実行し、
充填パターンの変化量から実際の転送バイト数を特定した。

| BX値 | 転送バイト数 | バッファの状態 |
|------|------------|-------------|
| 2    | 2バイト    | 先頭2バイトのみ変化、残りは0xAA |
| 512  | 512バイト  | 1セクタ全体が変化 |

#### 結論

**BX はバイト数を直接指定する。** `BX=512` で1セクタ（512バイト）を転送する。

PC-9800 Bible内のHDD BIOSサンプルコードでも `regs.r_bx = BYTE`（バイト数直接）が
使用されており、仕様テキストの「データブロック数」という表現は誤解を招く。

#### 修正

```asm
;; 修正前 (2バイトしか読まない)
mov     bx, 2

;; 修正後 (1セクタ=512バイト)
mov     bx, 512
```

### A-2. INT 1Bh のセクタ番号 (DL) は0開始

#### 問題

FDD BIOS（INT 1Bh AH=56h/76h）ではセクタ番号が1開始であるため、
HDD BIOSでも1開始と仮定して実装していた。

#### 実測結果

NHDファイルの既知セクタ（0, 3, 4, 5, 6, 7）にマーカーデータを書き込み、
INT 1Bh で DL=0〜10 をスキャンした結果:

| DL値 | NHDセクタ | データ |
|------|----------|-------|
| 0    | セクタ0  | IPL先頭 `EB 06 90 90` ✓ |
| 3    | セクタ3  | マーカー `SEC03!` ✓ |
| 4    | セクタ4  | マーカー `SEC04!` ✓ |
| 5    | セクタ5  | マーカー `SEC05!` ✓ |

#### 結論

**HDD BIOS の DL（セクタ番号）は 0 開始。** NHDファイルオフセットとの対応は:

```
file_offset = header_size + (C × heads × spt + H × spt + S) × sector_size
```

ここで S は 0 開始。LBA→CHS変換で `LBA % spt` がそのまま DL に使える。

### A-3. GDT エントリサイズの不一致

#### 問題

GDTディスクリプタを `dw` × 6 = 12バイトで定義していた。
正しくは各エントリ8バイトでなければならない。

```asm
;; 誤った定義 (12バイト/エントリ)
dw      0FFFFh, 0, 0, 09Ah, 0CFh, 0
```

これにより以下のバイト列が生成される:

```
FF FF 00 00 00 00 9A 00 CF 00 00 00
                  ↑ byte 5 = 0x00 (アクセスバイト = 存在しないセグメント!)
```

#### 修正

```asm
;; 正しい定義 (8バイト/エントリ)
dw      0FFFFh          ;; limit[15:0]
dw      0               ;; base[15:0]
db      0               ;; base[23:16]
db      09Ah            ;; access byte (Present, Ring0, Code, Exec+Read)
db      0CFh            ;; flags(4KB gran, 32bit) + limit[19:16]=Fh
db      0               ;; base[31:24]
```

### A-4. テスト用 p8 関数の ROL バグ

#### 問題

8ビット値をHEX 2桁で表示する関数 `p8` で、値を DL に格納したまま
`ROL DX, 4` を2回実行していた。ROL は DX 全体（16ビット）を回転するため、
DH（常に0）のニブルが先に出力され、**全ての値が "00" と表示された。**

#### 影響

全ての診断テスト（READテスト、セクタスキャン）の結果が "00" と表示され、
「INT 1Bh がデータを読めていない」と誤認。実際にはデータは正しく読めていた。

このバグの発見に最も多くの時間を費やした。

#### 修正

```asm
;; 修正前: 値がDLにあるのにDH(=0)のニブルが先に出る
xor     ah, ah
mov     dx, ax
rol     dx, 4

;; 修正後: 値をDHに置いてROL
mov     dh, al
xor     dl, dl
rol     dx, 4
```

### A-5. BIOS ワークエリアの破壊

#### 問題

ループカウンタを `0000:0400h`（物理アドレス 0x400）に配置したところ、
INT 1Bh 呼び出しのたびにカウンタが 0 にリセットされた。

#### 原因

PC-98 では `0000:0400h` 以降は **BIOS データ領域 (BDA)** であり、
INT 1Bh がこの領域をワークとして使用する。

#### 対策

ループカウンタはスタック上で管理するか、BDA外のRAM領域を使用する。

| 項目 | 値 | 備考 |
|------|-----|------|
| DA/UA | `80h` | SASI/IDE互換アドレス |
| SENSE (AH=84h) | 正常動作 | ジオメトリ取得可 |

## Part B: HDDディスクレイアウトとブートシーケンス

### B-1. ディスクジオメトリ

```
NHDイメージ仕様:
  シリンダ: 75
  ヘッド:   8
  セクタ/トラック: 17
  セクタサイズ:    512バイト
  セクタ/シリンダ: 8 × 17 = 136
  総容量: 75 × 136 × 512 = 5,222,400バイト (≈5MB)
```

### B-2. セクタレイアウト

```
LBA   内容                          サイズ
──────────────────────────────────────────────
  0   IPL (boot_hdd.bin)            512B (1セクタ)
  1   パーティションテーブル         512B (1セクタ)
  2-5 第2ステージローダー            2048B (4セクタ)
      (loader_hdd.bin)
  6-  カーネル (kernel.bin)          ≈80KB (≈157セクタ)
      ※ LBA 6 + ceil(kernel_size/512) まで
──────────────────────────────────────────────
 272  ext2パーティション開始          シリンダ2から
      (スーパーブロック: LBA 274-275)
      ※ LBA 272 = 2シリンダ × 136セクタ/シリンダ
──────────────────────────────────────────────
```

### B-3. パーティションテーブル (LBA 1)

PC-98形式のパーティションエントリ（16バイト/エントリ）:

| オフセット | 内容 | 値 |
|-----------|------|-----|
| 0 | ブータブルフラグ | `0x80` |
| 1 | システムタイプ | `0xE2` (ext2) |
| 6 | 開始セクタ | 0 |
| 7 | 開始ヘッド | 0 |
| 8-9 | 開始シリンダ | 2 |
| 10 | 終了セクタ | 16 (SPT-1) |
| 11 | 終了ヘッド | 7 (HEADS-1) |
| 12-13 | 終了シリンダ | 74 (CYL-1) |

### B-4. ブートシーケンス

```
BIOS → IPL (LBA 0, 1FC0:0000) → ローダー (LBA 2-5, 0000:8000)
  → カーネルロード (LBA 6-, 0900:0000)
  → A20ゲート有効化
  → GDTロード → CR0.PE=1 → far jmp 0008h:pm_inline32
  → プロテクトモード (セグメント初期化, スタック設定)
  → メモリプロービング (1MB以降, 512KB刻み)
  → BSSゼロクリア (kentry.asm)
  → kernel_main() へジャンプ
```

## Part C: HDDブート実現までに修正したバグ一覧

### C-1. カーネルとext2パーティションの重複 (致命的)

#### 症状

`MOUNT... root panic` でカーネルがハング。ext2スーパーブロックが破壊されていた。

#### 原因

カーネルバイナリ (≈80KB = 155-157セクタ) はLBA 6から開始し、LBA 160付近まで到達する。
しかしext2パーティションはシリンダ1 (LBA 136) から開始していたため、
**LBA 136-160の25セクタでカーネルデータとext2データが重複**していた。

```
カーネル:  LBA 6 ────────────────── LBA 160
ext2 (旧): LBA 136 ─────────────────────── LBA 10199
                    ^^^^^^^^^^^^^^^^^^^^^^^^
                    重複領域 (25セクタ)
```

特に `tools/write_ipl.py` でカーネルだけを後から書き込むと、
既存のext2スーパーブロック (LBA 138-139) が完全に破壊された。

#### 修正

パーティション開始をシリンダ1 (LBA 136) → **シリンダ2 (LBA 272)** に移動。

| ファイル | 変更内容 |
|---------|---------|
| `programs/install.c` | `HDD_PARTITION_LBA` = 272, パーティションテーブルの開始CYL=2 |
| `fs/ext2_super.c` | `ext2_base_lba` デフォルト値 = 272, フォールバック = 272 |
| `tools/install_hdd.py` | `PARTITION_LBA` = 272, パーティションエントリCYL=2 |

### C-2. install_hdd.py の間接ブロック未設定

#### 症状

小さなファイル (< 12KB) は読めるが、shell.bin (13KB, 14ブロック) や
vz.bin (49KB, 49ブロック) が読み込み時に不完全。

#### 原因

ext2ファイルシステムでは直接ブロックポインタは12本まで。13ブロック以上のファイルには
**間接ブロック** (i_block[12]) が必要だが、`install_hdd.py` は `blocks[:12]` しか
iノードに書き込んでいなかった。

```python
# 修正前: 12ブロックまでしか書き込まない
for bi, bp in enumerate(blocks[:12]):
    struct.pack_into('<I', fi, 40 + bi * 4, bp)
# TODO: 大きなファイルの間接ブロック対応
```

#### 修正

`install_hdd.py` に間接ブロック割り当て・書き込みロジックを追加:

1. 12ブロック以上のファイルに対して間接ブロックを割り当て
2. `i_block[12]` に間接ブロック番号を設定
3. 間接ブロック内に残りのデータブロックポインタを書き込み
4. ブロックビットマップと `i_blocks` カウントを更新

### C-3. ext2シングルトンのサブマウント破壊

#### 症状

`MOUNT... root OK` の後、`vfs_read("/shell.bin")` が `-2` (NOT FOUND) を返す。
ext2マウントは成功しているはずなのにファイルが見つからない。

#### 原因

カーネル初期化のサブマウントループが、ルート以外のブロックデバイス (例: hd1) に対して
`vfs_mount(mnt, dname, "ext2")` を呼び出す。ext2ドライバはシングルトン設計
（グローバル変数1セット）のため、`ext2_mount()` の先頭で:

```c
if (ext2_mounted) ext2_unmount();  /* ← ルートのext2が解除される！ */
```

が実行され、**ルートマウント済みのext2が破壊**された。hd1が存在しないので
新しいマウントも失敗 (`EXT2_ERR_IO`) し、結果として ext2 は未マウント状態に。

#### 修正

`kernel.c` のサブマウントループで、ルートがext2の場合は他のデバイスへの
ext2マウントをスキップするように変更:

```c
int root_is_ext2 = (root_fs[0] == 'e');
if (!root_is_ext2) {
    if (vfs_mount(mnt, dname, "ext2") == VFS_OK) mounted = 1;
}
```

### C-4. kentry.asm の BSSゼロクリア欠如

#### 症状

`exec: max nest level reached` が表示され、`exec_run()` が呼び出し前に拒否される。
`exec_nest_level` は初期値0のはずだが、4以上の不定値が入っていた。

#### 原因

`exec_nest_level` はBSSセクション（ゼロ初期化グローバル変数）に配置される。
通常のOS上ではローダーやCRT0がBSSをゼロクリアするが、ベアメタルの
`kentry.asm` は単に `jmp kernel_main` するだけで **BSSクリアを一切行っていなかった**。

```
.text: 0x09000 - 0x14E07 (コード)
.data: 0x14E20 - 0x1C853 (初期値あり変数)
.bss:  0x1C860 - 0x2C40B (ゼロ初期化変数) ← メモリ上はゴミデータのまま！
```

#### 修正

`kentry.asm` にBSSゼロクリアを追加:

```asm
kentry:
        ;; BSS領域ゼロクリア
        mov     edi, __bss_start
        mov     ecx, __bss_end
        sub     ecx, edi
        shr     ecx, 2
        xor     eax, eax
        rep     stosd

        jmp     kernel_main
```

リンカスクリプト (`os32.ld`) にはすでに `__bss_start` / `__bss_end` シンボルが
定義されていたため、それを利用。

## Part D: ツールとインストール

### D-1. tools/install_hdd.py — 直接インストーラー

ホスト側で直接NHDイメージを構築するPythonスクリプト。
OS32のゲストインストーラー (`programs/install.c`) を使わず、
ブートセクタ・ローダー・カーネル・ext2ファイルシステムを一括で書き込む。

```bash
python3 tools/install_hdd.py /tmp/test_new.nhd
```

主な機能:
- NHDヘッダ・ジオメトリの自動検出/生成
- IPL、パーティションテーブル、ローダー、カーネルの正確なLBA配置
- ext2ファイルシステムの構築（スーパーブロック、iノード、間接ブロック対応）
- `programs/*.bin` と `data/` ファイルの自動収集・小文字化

### D-2. tools/write_ipl.py — セクタ単位書き込み

既存NHDイメージの特定LBAにバイナリを上書きするツール。
カーネルのみの更新に使用。

```bash
python3 tools/write_ipl.py kernel.bin --sector 6
```

**注意**: カーネルサイズがパーティション開始LBA (272) を超えないことを確認すること。
現在のカーネルサイズ制限: **(272 - 6) × 512 = 136,192バイト (≈133KB)**

## Part E: 診断手法

### バッファ充填テスト

1. バッファを既知パターン（0xAA等）で充填
2. BIOS呼び出し実行
3. バッファの変化を確認

これにより「BIOSが実際にデータを転送したか」と「転送バイト数」を特定可能。

### マーカースキャン

1. ディスクイメージの既知セクタにユニークマーカーを書き込み
2. BIOS READで複数のCHS値をスキャン
3. マーカーが読めたCHS値からアドレッシングルールを確定

### TVRAM直接書き込みデバッグ

カーネルデバッグでは `kprintf` や `shell_print` のカーソル位置問題を回避するため、
TVRAMアドレス (`0xA0000` / `0xA2000`) に直接書き込む方法が有効:

```c
static void dbg_tvram(int x, int y, const char *s, u8 attr) {
    volatile u16 *text = (volatile u16 *)0xA0000;
    volatile u8  *atr  = (volatile u8  *)0xA2000;
    int i = y * 80 + x;
    while (*s) {
        text[i] = (u16)(u8)*s;
        atr[i*2] = attr;
        i++; s++;
    }
}
```
