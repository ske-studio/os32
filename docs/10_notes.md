
## 第10部 既知の制約と注意事項

### §10-1 DMA 64KB境界

- FDD読み書きバッファが64KB境界をまたいではならない (§2-2参照)
- loader.asmでは0x10000以降をES:BP=seg:0000方式で回避
- disk.cのDMAバッファはBSS配置で自動的に安全な配置になる

### §10-2 IPL (boot.asm) の制約

- NP21/Wエミュレータにおいて、IPL内でのINT 1Bh呼び出し回数に制約がある
- 5回以上のINT 1Bhを含むIPLではブートメッセージすら表示されない現象を確認
- **対策**: IPLは最小限のディスクI/O (4トラック=31セクタ) に留め、追加読み込みは
  loader.asm (第2段階ローダ) で行う二段階ブート方式を採用

### §10-3 BIOSトランポリン

- 32ビットPMからBIOS (16ビットRM) を呼ぶにはPM→RM→PM遷移が必要
- トランポリンコードは0x7000に配置 (リアルモードアクセス可能領域)
- 0x0600にBIOS_Regs構造体のミラーを配置
- loader.asmの拡張読み込みでも0x0600-0x0603をループ変数に一時使用

### §10-4 EGC farポインタ問題 (歴史的注記)

※過去にOpen Watcomの16-bitスモールモデルを使用していた際の問題であり、現在のGCC (32-bit プロテクトモードフラットモデル) 環境では発生しません。
かつては、EGCアクティブ中に2つの別々のfarポインタでVRAMにアクセスすると、コンパイラのセグメントレジスタ再ロードコードがEGCパイプラインに干渉していました。

### §10-5 INT 1Bh 呼び出し時の BIOS ワークエリア (BDA) 破壊

- PC-98では `0000:0400h` 以降は **BIOS データ領域 (BDA)** として予約されている
- INT 1Bh は内部的にこの領域をワークメモリとして使用する
- **対策**: ローダー開発時、ループカウンタや一時変数をスタックや `0x400` より前の安全なアドレスに避難させる必要がある (`0x400` に変数を置くと壊れる)

### §10-6 ext2 シングルトン構造によるサブマウント破壊 (解決済み)

- ~~現在の ext2 ドライバはシングルトン（グローバル変数1セットのみ）で実装されていた~~
- ~~`vfs_mount()` を別デバイスに対して呼び出すと、既存のルートマウント済み ext2 が強制的にアンマウントされる問題があった~~
- **対策完了**: VFSマルチインスタンスリファクタリング (v0.2) により、全FSドライバ (ext2/FAT12/SerialFS/ISO9660) がコンテキスト渡し方式 (Ext2Ctx等をkmalloc/kfree) に移行。複数デバイスの同時ext2マウントが可能になった。

### §10-7 kentry における BSS ゼロクリアの必要性

- 通常の OS や CRT0 が行う BSS のゼロクリアは、ベアメタルの場合自作する必要がある
- `exec_nest_level` などのグローバル変数が不定値（ゴミデータ）で初期化され、起動直後に動作不良を起こす一因となった
- **対策**: リセット直後の `kentry.asm` 内で `rep stosd` を用いて BSS セクション全体を自前でゼロクリアする実装が必須

### §10-8 カーネル成長による ext2 パーティションの重複

- カーネルバイナリ (約80KB) が LBA 6 から配置されるが、パーティションの開始 LBA が近すぎるとカーネル後部が ext2 スーパーブロック (通常 LBA 2付近) に上書きされてしまう問題があった
- **対策**: HDD NHDイメージ作成時、ext2の開始は シリンダ 2 (LBA 272) に移動しており、カーネルには最大およそ 133KB までの成長猶予が設定されている

### §10-9 プログラム終了時のリソース自動クリーンアップ

- 外部プログラムがclose/freeを忘れても、`exec_exit()` 内でカーネルが自動的に回収する
- **対象**: FDリダイレクト (stdin/stdout/stderr)、ファイルFD (FD 3以上)、パイプバッファ、共有メモリ
- **理由**: クラッシュ復帰時やユーザプログラムのバグによるリソースリークを防止するため
